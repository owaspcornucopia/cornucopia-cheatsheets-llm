# C8 — Insecure Infrastructure Configuration

## Threat

An attacker can bypass the application to gain access to data because the network and host infrastructure have not been securely configured.

## How This Applies

The deployment infrastructure is configured without security hardening:

- **Container runs without security restrictions** — the Docker Compose file does not set `read_only`, `no-new-privileges`, `cap_drop`, or security profiles
- **No network segmentation** — the application port is exposed directly without a firewall or network policy
- **No TLS proxy** — traffic is unencrypted between clients and the container
- **Database on local filesystem** — the SQLite file sits in the container with no access restrictions
- **Infrastructure is "fire and forget"** — deployed once without ongoing security patching, configuration review, or monitoring
- **No health checks** — no mechanism to detect if the application is compromised or behaving abnormally

The Docker Compose configuration allocates significant resources (8 CPUs, 12GB RAM) but applies no security constraints.

## Example Attack

An attacker exploits a vulnerability in one of the application's dependencies to gain code execution inside the container. Because the container has no security restrictions (no read-only filesystem, no dropped capabilities, no seccomp profile), the attacker has full control within the container — they can read the database, access model files, and potentially escape to the host.

## Mitigations

1. **Apply container security best practices** — set `read_only: true`, `security_opt: [no-new-privileges:true]`, and drop all unnecessary Linux capabilities.
2. **Run as non-root** — add `user:` directive to the Docker Compose service.
3. **Implement network policies** — restrict which services can communicate with each other.
4. **Add a TLS-terminating reverse proxy** in front of the application.
5. **Enable health checks** and monitoring to detect compromise or abnormal behavior.
6. **Implement regular patching** — rebuild containers with updated base images on a schedule.
7. **Use a read-only root filesystem** with explicit writable mounts only where needed.
