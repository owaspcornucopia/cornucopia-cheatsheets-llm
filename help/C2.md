# C2 — Dangerous Programming Patterns

## Threat

An attacker can bypass application controls because dangerous programming language functions have been used instead of safer alternatives.

## How This Applies

The application uses several dangerous programming patterns:

- **`conn.execute(query)` with unsanitized input** — direct SQL execution of an untrusted string, the classic dangerous function that should be replaced with parameterized queries
- **`ast.literal_eval()`** — used to parse LLM output when JSON parsing fails. While safer than `eval()`, it still processes untrusted input through a parser
- **`json.loads()` on LLM output** — processes untrusted model output without size limits or schema validation
- **String formatting in SQL inserts** — the `setupDB()` function uses f-strings to construct SQL (`f"INSERT ... values {row}"`) instead of parameterized queries

The most critical dangerous pattern is the direct execution of LLM-generated SQL. This is fundamentally equivalent to using `eval()` on user input — the application takes untrusted strings and executes them as code (SQL code, in this case).

## Example Attack

The `parse_tool_call()` function uses `ast.literal_eval()` as a fallback when JSON parsing fails. An attacker crafts LLM output that exploits this parser — while `literal_eval` is safer than `eval`, it still processes arbitrary Python literal syntax from an untrusted source.

## Mitigations

1. **Replace `conn.execute(query)` with parameterized queries** — never pass untrusted strings directly to the database engine.
2. **Remove `ast.literal_eval()`** — enforce strict JSON-only output from the LLM and reject anything that doesn't parse as valid JSON.
3. **Use parameterized SQL in `setupDB()`** — replace f-string interpolation with proper parameter binding.
4. **Add schema validation** on parsed JSON before using it — verify the structure matches expectations using a schema validator.
5. **Use the safest available function** for each operation — prefer parameterized queries, strict JSON parsing, and typed data handling.
