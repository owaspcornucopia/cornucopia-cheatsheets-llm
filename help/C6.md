# C6 — Inconsistent Error Handling

## Threat

An attacker can bypass controls or extract information because error handling is missing, inconsistent, or does not deny access by default.

## How This Applies

The application handles errors differently depending on where they occur:

1. **Tool call generation failure** — returns a 500 response with the exception message in the `error` field.
2. **Invalid tool call parsing** — returns a 200 response (not an error status code) with the raw model output included.
3. **SQL execution failure** — returns a 500 response with both the exception message and the SQL query that failed.
4. **Final answer generation failure** — returns a 500 response with the exception message.

Problems with this approach:
- Successful-looking responses (HTTP 200) are returned for failure cases, confusing API consumers
- Different error paths expose different levels of detail
- Werkzeug logging is explicitly disabled (`log.disabled = True`), reducing visibility into issues
- Error responses include internal state (SQL queries, model output) that helps attackers refine their attacks

## Example Attack

An attacker sends crafted prompts and observes the error responses. When the tool call parsing fails, they receive the raw model output (HTTP 200), telling them exactly how close they are to generating a valid injection. They iteratively refine their prompt until the model produces a valid tool call containing malicious SQL.

## Mitigations

1. **Standardize error responses.** All error conditions should return appropriate HTTP status codes with a generic message and an error reference ID.
2. **Never include internal state in error responses.** SQL queries, raw model output, and exception details belong in server-side logs only.
3. **Re-enable logging** for security-relevant events. Disabling Werkzeug logging removes visibility into attacks.
4. **Implement consistent error handling** using Flask's `@app.errorhandler` to ensure all exceptions are caught and sanitized before reaching the client.
5. **Fail closed** — when an error occurs, deny the request rather than returning partial results.
