# AZ6 — Access to Unauthorized Data Despite Valid Entry Point Access

## Threat

An attacker can access data they do not have permission to, even though they have legitimate permission to use the endpoint.

## How This Applies

An authenticated user (valid token) is permitted to use the `/api/fraud` endpoint. However, having access to the endpoint does not mean they should see all data in the database. The application enforces no data-level authorization:

- A valid token grants access to the endpoint — that is correct
- But that same token also grants access to every record in the database — that is not correct
- The SQL queries run with full database privileges, not scoped to the caller's authorized data
- Through SQL injection (via prompt manipulation), users can access any table, not just investigations

The distinction between "you can use this API" and "you can see this specific data" does not exist in the application.

## Example Attack

A user legitimately authorized to investigate their own transactions asks: "Show me all records where fraud was detected." The LLM generates `SELECT * FROM investigations WHERE fraud_detected='true'` — returning fraud records for ALL customers, not just the requesting customer's transactions.

## Mitigations

1. **Separate endpoint access from data access** — authenticating to the endpoint should not automatically grant access to all data.
2. **Enforce data-level authorization** — filter all query results to only include records the caller is permitted to see.
3. **Add a tenant/owner column** to the investigations table and enforce mandatory filtering on every query.
4. **Use database views or row-level security policies** to enforce data isolation at the database layer.
5. **Validate that the SQL generated by the LLM** only accesses data within the caller's authorization scope before executing it.
