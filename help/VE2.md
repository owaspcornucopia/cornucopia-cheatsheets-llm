# VE2 — Information Leakage via Error Messages

## Threat

An attacker can gather information about the application's internal workings through verbose error messages returned by the API.

## How This Applies

The `/api/fraud` endpoint returns detailed error information in its JSON responses when things go wrong:

- Raw LLM output is included in error responses (`raw_output` field)
- SQL queries generated by the model are returned to the caller (`sql_query` field)
- Python exception messages are sent back verbatim (`error` field)
- The system prompt structure is revealed through the conversation flow

When the LLM fails to produce a valid tool call, the API responds with the full raw model output. This tells an attacker exactly how the model responds, what format it uses, and what the system prompt expects — all useful for crafting prompt injection attacks.

## Example Attack

An attacker sends a deliberately unusual question that causes the model to produce unexpected output. The API returns the raw model response, revealing the system prompt format and expected JSON schema. The attacker uses this information to craft a prompt injection that produces a valid-looking tool call containing malicious SQL.

## Mitigations

1. **Return generic error messages to callers.** Replace detailed error responses with a simple error ID and human-readable message. Log the details server-side only.
2. **Never return raw model output to the client.** If the model fails to produce a valid tool call, return a generic "unable to process your request" message.
3. **Strip SQL queries from error responses.** Internal query details should never be visible to API consumers.
4. **Use structured logging** to capture diagnostic information server-side where developers can access it without exposing it through the API.
