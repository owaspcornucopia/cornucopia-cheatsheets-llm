# C9 — Feature Misuse or Resource Overconsumption

## Threat

### Race Condition on Database Initialisation

`app.py` calls `setupDB()` at startup, which:
1. Deletes the existing database file with `os.remove(db_path)`
2. Recreates it from scratch with the schema and seed data

When the application is scaled (`--scale app=3`), all three instances start concurrently and each executes `setupDB()` simultaneously. The result is a race condition on the shared SQLite file:

- Instance A deletes the file; Instance B tries to execute on a file that no longer exists
- Instance A recreates the file; Instance B deletes it again
- The database ends up empty, partially written, or corrupted depending on timing
- Investigation data is silently destroyed without any error being surfaced to the user

Note: this is distinct from the LLM inference resource exhaustion covered by LLM2. LLM2 addresses unlimited inference requests overwhelming the model. C9 here addresses a startup race condition and the broader misuse surface created by the scaled, load-balanced architecture.

### Amplified Request Volume Against the Model Service

With nginx load-balancing across multiple `app` instances, an attacker with a valid token can send one request per `app` instance simultaneously. Since all app instances call the same `model_service`, concurrent long-running inference jobs can queue up and starve other users. The model service has no concurrency limit or queue depth protection.

## Attack Example

Mike starts the application with `--scale app=3`. All three instances race on startup to reinitialise the database. The seed data ends up inconsistent or entirely missing. When game participants make API calls, they receive unexpected or empty results, breaking the demo scenario.

Separately, Mike sends 3 simultaneous long questions (one per app instance) to `/api/fraud`. All 3 forward to `model_service` at once. The model service processes them serially (CPU-bound), and all responses are delayed by the queue — effectively a low-cost denial of service against other legitimate users sharing the service.

## Mitigations

### 1. Make setupDB Safe for Scaled Deployments (Critical)

Move database initialisation out of each app instance's startup into a dedicated one-time init step:

**Option A — Docker Compose init container** (already present as `downloader`; extend it):
```yaml
# In docker-compose.yml, have the app depend on a db-init service
db-init:
  build: .
  command: python -c "from app import setupDB; setupDB()"
  volumes:
    - db-data:/data
  environment:
    - DB_CONNECTION_STRING=/data/db.sqlite
```

**Option B — Guard with a file lock in setupDB**:
```python
import fcntl
def setupDB():
    lock_path = db_path + ".lock"
    with open(lock_path, "w") as lock_file:
        fcntl.flock(lock_file, fcntl.LOCK_EX)  # blocks until acquired
        if not os.path.exists(db_path):          # only init if not already done
            _create_db(db_path)
```

**Option C — Use `INSERT OR IGNORE` without `os.remove()`**:
Remove the `os.remove(db_path)` call. The existing `INSERT OR IGNORE` already handles re-runs safely, and the `CREATE TABLE IF NOT EXISTS` is already idempotent.

### 2. Add Concurrency Limits to the Model Service

Limit simultaneous in-flight requests to `model_service` using a semaphore:
```python
import threading
_inference_lock = threading.Semaphore(1)  # only 1 concurrent inference

@app.route('/generate', methods=['POST'])
def generate():
    if not _inference_lock.acquire(blocking=False):
        return jsonify({"error": "Model is busy, try again shortly"}), 503
    try:
        result = generate_once(messages)
        return jsonify({"result": result})
    finally:
        _inference_lock.release()
```

### 3. Add Rate Limiting at nginx

```nginx
http {
    limit_req_zone $http_token zone=per_token:10m rate=2r/m;

    server {
        location /api/fraud {
            limit_req zone=per_token burst=2 nodelay;
            proxy_pass http://app_backend;
        }
    }
}
```
This limits each token to 2 requests per minute, preventing amplified abuse through multiple app instances.
